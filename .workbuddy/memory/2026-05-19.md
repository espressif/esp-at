## 完成了 esp-at 项目整体架构阅读与分析

阅读了以下核心文件，梳理了完整架构：
- main/app_main.c：程序入口
- main/interface/at_interface_api.c：接口抽象层
- main/interface/uart/at_uart_task.c：UART 驱动与任务
- components/at/src/at_init.c：AT 框架初始化
- components/at/src/at_cmd_register.c：AT 命令注册机制
- components/at/include/esp_at_core.h：AT 核心数据结构
- components/at/include/esp_at_cmd_register.h：命令注册宏
- components/at/src/at_user_cmd.c：用户自定义命令示例

**架构总结**：ESP-AT 是分层式 AT 固件框架，从底往上分为：
1. 物理接口层（UART/SPI/SDIO/Socket）
2. 接口抽象层（at_interface_api.c 统一 read/write/ops）
3. AT 核心框架（闭源 .a 库，负责命令解析、任务调度）
4. AT 命令集（按 CONFIG 宏条件编译，通过 linker section 自动注册）
5. 应用入口（app_main.c → esp_at_init()）

## WiFi 与 BLE 共存机制分析

**核心结论：ESP-AT 框架本身不支持 WiFi + BLE 同时开启，官方模块配置均为互斥。**

关键发现：
- `module_esp32c2_default`（4MB）：BT_ENABLED=y，AT_BLE_COMMAND_SUPPORT=**n**，即蓝牙硬件初始化但 AT 命令集未注册（内存释放后仅剩 WiFi AT 用途）
- `module_esp32c2-2mb-ble`：AT_BLE_COMMAND_SUPPORT=y，AT_WIFI_COMMAND_SUPPORT=**n**，纯 BLE 模式
- `module_esp32c2-2mb`：AT_BLE_COMMAND_SUPPORT=y + WIFI 支持 + SW_COEXIST=y，是唯一 WiFi+BLE 共存配置
- 所有纯 BLE 模块：`CONFIG_ESP_COEX_SW_COEXIST_ENABLE=n`，纯 WiFi 模块也=n
- WiFi+BLE 共存模块：`CONFIG_ESP_COEX_SW_COEXIST_ENABLE=y`（必须开启软件共存）
- `at_bt_controller_mem_release()`：若只用 BLE，会释放 Classic BT 内存；若两者都不用，释放全部 BT 内存

**处理机制隐患**（基于历史调试经验）：
- MH2101 主机侧通过单 UART 通道同时控制 WiFi AT 和 BLE AT，需要上层互斥锁（Set_Wifi_BT_Used_Status）
- BLE_Reset() 切换到 BLE Server 模式时需要硬件断电重启 ESP32，WiFi 会断开
- BLE_SendDataToNotify 超时问题：数据实际已发出，响应解析时序不匹配
